## 1. Контекст

Разреженные матрицы (содержащие преимущественно нули) требуют специальных форматов хранения
и алгоритмов, чтобы избежать обработки нулевых элементов. В данной работе используется формат
CCS (Compressed Column Storage), который хранит только ненулевые значения и их индексы строк,
а также массив указателей на начала столбцов.

Цель работы — реализовать умножение двух разреженных матриц с комплексными элементами в формате
CCS, используя различные технологии параллельного программирования: последовательный алгоритм
(SEQ), OpenMP, Intel TBB, стандартную библиотеку C++ (STL) с потоками, а также гибрид MPI+OpenMP.
Все реализации сравниваются по производительности и эффективности масштабирования.

## 2. Постановка задачи

- **Вход:** Две матрицы `A` (размера `M×K`) и `B` (размера `K×N`) в формате CCS.
- **Выход:** Матрица `C = A × B` (размера `M×N`) в формате CCS.
- **Условие корректности:** `A.count_cols == B.count_rows`.
- **Элементы:** `std::complex<double>`.
- **Порог отбрасывания:** Элементы, чей модуль меньше `1e-15`, считаются нулями.
- **Проверка корректности:** Сравнение с эталонным плотным умножением (GTest).

## 3. Базовый алгоритм (последовательная версия)

Все параллельные реализации основаны на едином последовательном алгоритме, который использует
подход «столбец за столбцом»:

1. Выделяется плотный вектор-аккумулятор `dense_col` размера `M` и массив маркеров `is_active`.
2. Для каждого столбца `j` матрицы `B`:
   - Для каждого ненулевого элемента `B_{k,j}` из столбца `j`:
     - Извлекается столбец `k` матрицы `A` (все `A_{i,k}`).
     - Каждый элемент `A_{i,k}` умножается на `B_{k,j}` и прибавляется к `dense_col[i]`.
   - Активные строки (где `dense_col[i] != 0`) сохраняются в `active_rows`.
   - Сортировка `active_rows` (по возрастанию индексов).
   - Ненулевые результаты переносятся в выходные массивы `values` и `row_index`.
   - Очистка `dense_col` и `active_rows`.
3. Формируется массив `col_index` для выходной матрицы.

Асимптотика в худшем случае (плотные матрицы): `O(M * K * N)`, но для разреженных матриц
сложность определяется количеством ненулевых элементов: `O(nnz(A) * nnz(B) / K)`.

## 4. Реализации и схемы распараллеливания

### 4.1. Последовательная (SEQ)

Без изменений — используется описанный выше алгоритм. Служит эталоном для проверки
корректности и измерения ускорения.

### 4.2. OpenMP (OMP)

**Схема распараллеливания:** Параллелизуется внешний цикл по столбцам `B` (каждый столбец `C`
вычисляется независимо). Каждый поток создаёт собственные локальные буферы (`accumulator`,
`active_rows`, `row_marker`), что исключает гонки данных.

```cpp
#pragma omp parallel default(none) shared(...)
{
    // локальные векторы
    #pragma omp for schedule(dynamic)
    for (int col_idx_b = 0; col_idx_b < cols_b_count; ++col_idx_b) {
        ProcessSingleColumn(...);
    }
}
```

**Особенности:** Использован `schedule(dynamic)` для балансировки нагрузки, так как плотность
столбцов может сильно различаться. Финальная конкатенация результатов выполняется
последовательно.

### 4.3. Intel TBB (TBB)

**Схема распараллеливания:** Задачно-ориентированный подход. Вместо линейной комбинации
столбцов применяется скалярное произведение с предварительным транспонированием матрицы `A`.
Это позволяет эффективно использовать `tbb::parallel_for`:

```cpp
tbb::parallel_for(tbb::blocked_range<int>(0, res_cols),
    [&](const tbb::blocked_range<int> &r) {
        for (int j = r.begin(); j != r.end(); ++j) {
            ProcessColumn(j, ...);
        }
    });
```

**Особенности:** TBB автоматически разбивает диапазон на блоки (work stealing).
Каждый столбец хранит свои векторы результатов, затем данные объединяются через
`std::make_move_iterator` (move-семантика). Это минимизирует копирование.

### 4.4. STL (std::thread)

**Схема распараллеливания:** Декомпозиция по данным — по ненулевым элементам матрицы `A`.
Матрица `B` транспонируется. Нагрузка (индексы элементов `A`) разбивается на `num_threads`
диапазонов. Каждый поток обрабатывает свой диапазон, накапливая произведения в локальный
`std::map< pair<col, row>, complex<double>>`. После завершения всех потоков словари
сливаются в один глобальный.

```cpp
std::vector<std::thread> threads;
for (int i = 0; i < num_threads; ++i) {
    threads.emplace_back(ProcessChunk, start, end, ...);
}
```

**Особенности:** Использование `std::map` обеспечивает автоматическую сортировку по столбцам,
но создаёт значительные накладные расходы на динамическую аллокацию памяти.

### 4.5. Гибрид MPI + OpenMP (ALL)

**Схема распараллеливания:** Двухуровневая. MPI распределяет столбцы результирующей матрицы
между процессами циклически (`idx_j = rank; idx_j < N; idx_j += size`). Внутри каждого
MPI-процесса столбцы обрабатываются с помощью OpenMP (циклы по строкам `A`, скалярное
произведение). После локальных вычислений выполняется сборка глобального результата:
`MPI_Allreduce` для суммирования количества ненулевых элементов, затем `MPI_Bcast`
для рассылки данных столбцов всем процессам (требование `kALL`).

```cpp
// Внутри MPI-процесса
#pragma omp parallel for schedule(dynamic)
for (int idx_j = rank; idx_j < mat_b.count_cols; idx_j += size) {
    for (int idx_i = 0; idx_i < mat_a.count_rows; ++idx_i) {
        res = DotProduct(mat_at, idx_i, mat_b, idx_j);
        // сохранение в локальные векторы
    }
}
// MPI сборка
```

**Особенности:** Комплексные числа передаются через два массива `double` (real/imag).
Циклическое распределение обеспечивает лучшую балансировку, чем блочное.

## 5. Детали реализации (общие черты)

- **Валидация формата CCS:** Все версии проверяют, что `col_index` монотонно возрастает,
  индексы строк находятся в допустимых пределах и т.д. (метод `IsValidCCS`).
- **Обработка нулевых матриц:** Если `nnz_A == 0` или `nnz_B == 0`, результат — нулевая
  матрица, заполняются только `col_index`.
- **Эпсилон:** При сравнении результатов используется `eps = 1e-12` (для комплексных чисел
  сравниваются действительная и мнимая части по модулю).
- **Тестирование:** 6 функциональных тестов (единичная матрица, прямоугольные матрицы,
  нулевая, только мнимые числа, случайные разреженные). Производительность замеряется
  на матрицах 1000×1000 с плотностью 1% (около 10000 ненулевых элементов).

## 6. Проверка корректности

Все реализации прошли функциональные тесты. Результаты параллельных версий побитово
(с учётом `eps`) совпадают с последовательным эталоном. Детерминизм обеспечен
за счёт того, что каждый поток пишет в свои локальные структуры, а финальная сборка
происходит в строго фиксированном порядке.

## 8. Результаты производительности

Тестовые матрицы: две случайные разреженные матрицы 1000×1000, плотность 1%,
элементы – комплексные числа от -10 до 10. Время измерялось для `Task_run` (основное
вычисление) в миллисекундах.

| Технология | Конфигурация          | Время (мс) | Ускорение | Эффективность |
|------------|-----------------------|------------|-----------|---------------|
| SEQ        | 1 поток               | 56.1       | 1.00x     | 100%          |
| OMP        | 2 потока              | 31.8       | 1.76x     | 88.0%         |
| OMP        | 4 потока              | 18.5       | 3.03x     | 75.7%         |
| TBB        | 2 потока              | 30.2       | 1.86x     | 93.0%         |
| TBB        | 4 потока              | 15.9       | 3.53x     | 88.2%         |
| STL        | 2 потока              | 41.5       | 1.35x     | 67.5%         |
| STL        | 4 потока              | 33.2       | 1.63x     | 40.7%         |
| MPI+OMP    | 2x1 (2 MPI)           | 35.4       | 1.58x     | 79.0%         |
| MPI+OMP    | 2x2 (всего 4)         | 21.5       | 2.61x     | 65.2%         |
| MPI+OMP    | 1x4 (только OMP)      | 18.5       | 3.03x     | 75.7%         |

**Анализ:**

- **TBB** показал наилучшее ускорение (3.53x на 4 потоках), благодаря эффективному
  планировщику work-stealing и использованию скалярного произведения с транспонированием.
- **OpenMP** также хорош (3.03x), но уступает TBB из-за последовательной фазы конкатенации.
- **STL** отстаёт из-за накладных расходов `std::map` и последовательного слияния словарей.
- **Гибрид MPI+OMP** на одном узле проигрывает чистому OpenMP из-за коммуникационных
  затрат (`MPI_Bcast`). Однако на кластерах он позволяет обрабатывать матрицы, не
  помещающиеся в память одного узла.

## 9. Выводы

1. **Масштабируемость:** Для 2–4 потоков на общей памяти лучшие результаты даёт TBB,
   затем OpenMP. STL неэффективен для данной задачи.
2. **Динамическая балансировка:** TBB (work-stealing) и OpenMP с `schedule(dynamic)`
   успешно справляются с неравномерной плотностью столбцов.
3. **Коммуникационные накладные расходы:** Гибрид MPI+OMP оправдан только для
   распределённых систем; на одном узле лучше использовать TBB или OpenMP.
4. **Практические рекомендации:**
   - Для плотности менее 5% на матрицах размером до 2000×2000 достаточно TBB или OpenMP.
   - Для сверхразреженных матриц (<<1%) предпочтителен TBB из-за низких накладных
     расходов на управление задачами.
   - При работе на кластере необходимо использовать гибрид MPI+OpenMP с циклическим
     распределением столбцов.
5. **Корректность:** Все реализации прошли функциональное тестирование и дают идентичные
   результаты.

Итоговая рекомендация: **TBB** — наилучший выбор для разреженных матриц в среде
общей памяти, обеспечивая высокую эффективность и простоту кода.
