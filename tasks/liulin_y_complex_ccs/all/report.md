## 1. Контекст

Гибридная реализация является кульминацией работы, объединяя две модели параллелизма:
распределенную память (MPI) для масштабирования между узлами и общую память (OpenMP)
для эффективного использования ядер внутри каждого узла. Задача типа `kALL` подразумевает,
что итоговый результат вычислений должен быть доступен на всех участвующих процессах.

## 2. Постановка задачи

- **Вход/Выход:** Две матрицы CCS `A` и `B`, результат `C = A * B` в формате CCS.
  Все данные — комплексные числа.
- **MPI-топология:** Вычисления распределяются между `size` процессами.
- **OpenMP-топология:** Внутри каждого MPI-процесса вычисления распределяются
  между доступными потоками.

## 3. Схема распараллеливания и алгоритм

Для обеспечения максимальной производительности в распределенной среде алгоритм
был разделен на несколько фаз:

### 1. Распределение входных данных (MPI)

Матрицы `A` и `B` изначально находятся на процессе с рангом 0.

- Используется функция `BroadcastMatrix`, которая через `MPI_Bcast` рассылает метаданные
  и массивы структуры всем процессам.
- Поскольку `std::complex<double>` не является примитивным типом MPI, значения передаются
  как два отдельных массива `double` (вещественная и мнимая части).

### 2. Подготовка и декомпозиция

- **Транспонирование:** Каждый процесс локально транспонирует матрицу `A` в `mat_at`.
  Это позволяет вычислять значения ячеек через скалярное произведение столбцов (Dot Product),
  что гораздо эффективнее для разреженных данных.
- **Циклическое распределение:** Столбцы результирующей матрицы $C$ распределяются
  между процессами **циклически**: `idx_j = rank; idx_j < count_cols; idx_j += size`.
  Это гарантирует лучшую балансировку нагрузки по сравнению с блочным распределением,
  так как плотность ненулевых элементов в разреженных матрицах часто распределена
  крайне неравномерно.

### 3. Внутриузловые вычисления (OpenMP)

Внутри каждого MPI-процесса цикл по выделенным ему столбцам распараллеливается:

```cpp
#pragma omp parallel for default(none) shared(rank, size, mat_a, mat_b, mat_at, local_rows, local_vals, local_nnz) \
    schedule(dynamic)
    for (int idx_j = rank; idx_j < mat_b.count_cols; idx_j += size) {
      for (int idx_i = 0; idx_i < mat_a.count_rows; ++idx_i) {
        std::complex<double> res = DotProduct(mat_at, idx_i, mat_b, idx_j);
        if (std::abs(res.real()) > kEpsilon || std::abs(res.imag()) > kEpsilon) {
          local_rows[static_cast<size_t>(idx_j)].push_back(idx_i);
          local_vals[static_cast<size_t>(idx_j)].push_back(res);
          local_nnz[static_cast<size_t>(idx_j)]++;
        }
      }
    }
```

Использование `schedule(dynamic)` позволяет потокам не простаивать, если один из столбцов
оказался значительно "тяжелее" (плотнее) остальных.

### 4. Сборка глобального результата (MPI)

Так как задача относится к типу `kALL`, необходимо собрать данные со всех процессов:

1. **`MPI_Allreduce`**: Суммирует локальные счетчики ненулевых элементов по каждому столбцу.
   После этого каждый процесс знает точный размер финальных массивов и структуру `col_index`.
2. **`GatherColumnData`**: Использует серию вызовов `MPI_Bcast`. Процесс-владелец столбца
   рассылает его данные (`row_index` и `values`) всем остальным. По завершении цикла
   каждый MPI-процесс владеет полной и идентичной копией матрицы `C`.

## 4. Особенности реализации

- **Dot Product:** Использование метода двух указателей для нахождения пересечения индексов
  в отсортированных столбцах CCS.
- **Обработка комплексных чисел:** Ручное разделение на Real/Imag части при MPI-коммуникациях
  для обеспечения переносимости и корректности типов данных.
- **Безопасность:** Проверка валидности структуры CCS выполняется на Rank 0 перед началом
  рассылок. Любые исключения в `RunImpl` перехватываются (`try-catch`), возвращая `false`.

## 7. Результаты производительности

Параметры теста: случайные разреженные матрицы 1000x1000 с плотностью заполнения 1%.

| Конфигурация (MPI x OMP) | Время Task_run (мс) | Ускорение | Эффективность |
| ------------------------ | ------------------- | --------- | ------------- |
| 1 x 1 (SEQ)              | 56.1                | 1.00x     | 100%          |
| 2 x 1 (Только MPI)       | 35.4                | 1.58x     | 79.0%         |
| 2 x 2 (Всего 4 ядра)     | 21.5                | 2.61x     | 65.2%         |
| 1 x 4 (Только OMP)       | 18.5                | 3.03x     | 75.7%         |

**Анализ результатов:**
Гибридная версия работает быстрее последовательной, но уступает чистому OpenMP
на одном узле. Это объясняется существенными накладными расходами на коммуникацию:
`MPI_Bcast` внутри цикла по столбцам создает большую нагрузку на шину данных
при пересылке разреженных векторов. Однако на распределенных системах (кластерах)
такая архитектура — единственный способ обработки матриц, не помещающихся
в память одного узла.

## 8. Выводы

1. **Масштабируемость:** Гибридный подход позволяет использовать вычислительные мощности
   нескольких узлов одновременно.
2. **Балансировка:** Циклическое распределение в сочетании с `dynamic schedule` в OpenMP
   минимизирует дисбаланс нагрузки.
3. **Коммуникационный оверхед:** Для разреженных матриц малого размера (1000x1000)
   затраты на MPI-обмены значительны. Гибридная реализация наиболее эффективна
   на сверхбольших разреженных структурах.
4. **Универсальность:** Код корректно работает на любых конфигурациях (от одного процесса
   до больших кластеров).
